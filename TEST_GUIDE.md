# Nearby Payments API ‚Äî Test Guide

## Prerequisites

- Server running locally: or `air` for live reload
- Postman with `nearby-api.postman_collection.json` imported
- `.env` loaded ‚Äî all environment variables set
- Confirm server is up: `GET http://localhost:8080/health` ‚Üí `{"status":"ok"}`

---

## Auth Levels

Every protected endpoint uses one of two auth levels enforced by middleware:

| Level | What it checks |
|-------|----------------|
| **low** | Valid Bearer token only |
| **high** | Bearer token + `X-Device-Provider`, `X-Request-Nonce`, `X-Request-Timestamp` headers. Nonce must be unique (replay protection via Redis). Timestamp must be within 5 minutes of now. |

The Postman pre-request scripts on `high` endpoints generate these headers automatically.

---

## Step 1 ‚Äî Get an Access Token

The fastest way is through the browser test page.

1. Open `http://localhost:8080/static/auth_test.html`
2. Select platform (`ios` or `android`) and click **Sign in with Google**
3. Complete Google sign-in
4. The page redirects back and completes the exchange automatically
5. Copy the **Access Token** from the green box

Paste the token into the Postman collection variable `accessToken`:
- Click the collection name ‚Üí **Variables** tab ‚Üí set `accessToken` current value

All requests that use `Authorization: Bearer {{accessToken}}` will now authenticate.

**What the sign-in creates in the database:**
- A `users` row
- A `sessions` row (access + refresh tokens)
- A `wallet_bindings` row (required for deposit endpoints)
- A `device_integrity` row

---

## Step 2 ‚Äî Health & Public Key

These require no auth token.

### Health Check
```
GET /health
```
Expected: `200 {"status":"ok"}`

### Get Server Public Key
```
GET /v1/auth/server-public-key
```
Expected: `200` with the server's ed25519 public key. Used by mobile clients to verify credential signatures.

---

## Step 3 ‚Äî Auth Endpoints

### Refresh Session
- Auth level: **low**
- The `refreshToken` collection variable is auto-populated after sign-in
- Send as-is; the test script saves the new `accessToken` and `refreshToken`

```
POST /v1/auth/refresh
Body: { "refreshToken": "{{refreshToken}}" }
```
Expected: `200` with new `accessToken` and `refreshToken`

**Note:** Each refresh token is single-use. After a successful refresh, the old token is invalid.

### Revoke Session
- Auth level: **low**
- Signs out the current session; the access token becomes invalid immediately

```
POST /v1/auth/revoke
```
Expected: `204 No Content`

After revoking, sign in again via the browser page to get a fresh token.

### Assert Device Integrity
- Auth level: **high** (headers auto-generated by pre-request script)
- Used by mobile clients after attestation. In testing, `provider: stub` bypasses real attestation.

```
POST /v1/auth/integrity
Body: { "deviceIntegrity": { "provider": "stub", "keyId": "" }, "timestampMs": <unix_ms> }
```
Expected: `200`

### Issue Device Credential
- Auth level: **high** (headers auto-generated)
- Issues a signed credential tied to the device's local proof key

```
POST /v1/auth/credential
Body: { "localProofPublicKey": "0x<64-hex-chars>" }
```
Expected: `200` with a signed credential

---

## Step 4 ‚Äî Deposit Endpoints

All deposit endpoints require **low** auth. They also require a wallet binding, which is created automatically during sign-in when `suiAddress` is included (the browser test page handles this).

If you get `422 Unprocessable Entity`, it means the wallet binding is missing ‚Äî sign out and sign in again through the browser page.

### Get Deposit Options
```
GET /v1/deposit/options
```
Expected: `200` with available deposit methods and your virtual account details (bank routing number, account number).

### Get Deposit History
```
GET /v1/deposit/history?limit=20&offset=0
```
Expected: `200` with a list of deposits (empty array if none yet)

Query params `limit` and `offset` are optional (default: 20, 0).

### Get Deposit by ID
- Set `depositId` collection variable to an ID from the history response first

```
GET /v1/deposit/{{depositId}}
```
Expected: `200` with deposit details, or `404` if ID doesn't exist / belongs to another user

## Step 5 ‚Äî Payment Endpoints

### Create Payment Intent
- Auth level: **high** (headers auto-generated)
- Requires `Idempotency-Key` header (auto-generated by pre-request script)
- The `idempotencyKey` must also appear in the request body ‚Äî copy the value from `{{_lastIdempotencyKey}}` (set automatically by the script) into the body field before sending

```
POST /v1/payments/intents
Headers: Idempotency-Key: <auto-generated>
Body:
{
  "recipientAddress": "0x<64-hex-chars>",
  "recipientName": "alice@nearby.sui",
  "asset": "USDsui",
  "amountAtomic": "1000000",
  "fundingMode": "sponsored",
  "idempotencyKey": "<same value as Idempotency-Key header>"
}
```

**Important constraints:**
- `asset` must be exactly `"USDsui"` ‚Äî any other value returns `400 asset_unsupported`
- `recipientAddress` must be a 66-character string starting with `0x` (0x + 64 hex chars)
- `amountAtomic` is in micro-units (`"1000000"` = 1 USDsui)
- `fundingMode`: `"sponsored"` (server pays gas) or `"user_paid"` (user pays gas)
- Intent expires in **2 minutes** ‚Äî submit or cancel before then

Expected: `201` with:
```json
{
  "intentId": "...",
  "recipientAddress": "...",
  "asset": "USDsui",
  "amountAtomic": "1000000",
  "fundingMode": "sponsored",
  "sponsorAddress": "0x...",
  "status": "pending",
  "expiresAt": <unix_seconds>
}
```

The test script saves the `intentId` to `{{intentId}}` automatically.

Resending with the same `Idempotency-Key` header returns `409 idempotent_replay` from the middleware. The body-level `idempotencyKey` allows the service to return the original intent if retried after the Redis key expires (24h window).

### Get Payment Intent
- Auth level: **low**

```
GET /v1/payments/intents/{{intentId}}
```
Expected: `200` with intent details including current `status` (`pending`, `submitted`, `cancelled`, `failed`)

### Cancel Payment Intent
- Auth level: **low**
- Only works if the intent is still in `pending` status

```
POST /v1/payments/intents/{{intentId}}/cancel
```
Expected: `204 No Content`

After cancellation, `GET /v1/payments/intents/{{intentId}}` returns `status: "cancelled"`.

### Submit Payment Intent
- Auth level: **low**
- Requires a real Sui transaction ‚Äî cannot be tested without a Sui wallet and on-chain funds
- The mobile app builds the transaction, signs it, and sends the bytes here

```
POST /v1/payments/intents/{{intentId}}/submit
Body:
{
  "txBytes": "<base64-encoded Sui transaction bytes>",
  "userSignature": "<Sui signature>"
}
```
Expected: `200` with `paymentId`, `txDigest`, `status: "confirmed"`

The test script saves `paymentId` to `{{paymentId}}`.

### Get Payment by ID
- Auth level: **low**

```
GET /v1/payments/{{paymentId}}
```
Expected: `200` with full payment details including `txDigest` and `confirmedAt`

## Step 6 ‚Äî Names Endpoints

Both require **high** auth (headers auto-generated).

### Register Leaf Name
Registers an on-chain SuiNS name (e.g. `alice.nearby`). Returns a task ID ‚Äî name registration is async.

```
POST /v1/names/leaf
Body: { "leafName": "alice", "parentName": "nearby" }
```
Expected: `202 Accepted` with `taskId`

The test script saves `taskId` to `{{taskId}}`.

**Note:** This triggers a real on-chain transaction. Ensure your SuiNS configuration is set up correctly in `.env`.

### Get Name Task
Poll this to check registration status.

```
GET /v1/names/tasks/{{taskId}}
```
Expected: `200` with `status` field (`pending`, `confirmed`, `failed`)

---

## Step 7 ‚Äî Nearby Sessions

All three require **high** auth (headers auto-generated).

### Initiate Session
Sender initiates a proximity payment session targeting a recipient's Sui address.

```
POST /v1/nearby/sessions
Body:
{
  "recipientSuiAddress": "0x<64-hex-chars>",
  "payloadType": "payment_request",
  "payloadData": "{\"amount\": \"1000000\"}"
}
```
Expected: `201` with session `id` and a connection token/payload for the recipient

The test script saves `id` to `{{sessionId}}`.

### Get Session
```
GET /v1/nearby/sessions/{{sessionId}}
```
Expected: `200` with session status and payload

### Acknowledge Session
Recipient accepts or rejects the session.

```
POST /v1/nearby/sessions/{{sessionId}}/acknowledge
Body: { "accept": true }
```
Expected: `200`

## Step 8 ‚Äî Webhooks

### Bridge Webhook
Called by the Bridge provider when a fiat deposit arrives. Not user-triggered ‚Äî included for completeness.

```
POST /v1/webhooks/bridge
Headers: X-Bridge-Signature: <HMAC signature>
Body: { "type": "virtual_account.activity", "data": {} }
```

Without a valid signature, the server will reject it. To test locally, you need a real event from the Bridge sandbox with a matching signature.

## Common Error Responses

| Status | Code | Cause |
|--------|------|-------|
| 400 | `bad_request` | Malformed JSON body |
| 400 | `validation_error` | Missing required field |
| 400 | `idempotency_key_required` | Missing `Idempotency-Key` header on payment intents |
| 400 | `asset_unsupported` | Asset is not `USDsui` |
| 400 | `invalid_address` | Recipient address not 66 chars / not `0x`-prefixed |
| 401 | `unauthorized` | Missing or invalid Bearer token |
| 401 | `high_fidelity_required` | Missing `X-Device-Provider`, `X-Request-Nonce`, or `X-Request-Timestamp` |
| 401 | `timestamp_out_of_window` | Timestamp is more than 5 minutes old |
| 401 | `replay_detected` | Nonce has been used before |
| 404 | `not_found` | Resource doesn't exist or belongs to another user |
| 409 | `idempotent_replay` | `Idempotency-Key` already used within 24 hours |
| 422 | `unprocessable` | Wallet binding missing ‚Äî sign in again with `suiAddress` |
| 500 | `internal_error` | Database or external service error |

## Recommended Test Sequence

```
1. Health Check
2. Get Server Public Key
3. Sign in via browser ‚Üí paste accessToken into Postman
4. Get Deposit Options          (confirms wallet binding exists)
5. Get Deposit History          (empty array expected)
6. Create Payment Intent        (note: asset = USDsui, idempotencyKey in body = header value)
7. Get Payment Intent           (status: pending)
8. Cancel Payment Intent        (status ‚Üí cancelled)
9. Create another Payment Intent (new Idempotency-Key generated automatically)
10. Get Payment Intent          (status: pending)
11. Refresh Session             (new tokens saved automatically)
12. Initiate Nearby Session
13. Get Nearby Session
14. Acknowledge Nearby Session
15. Register Leaf Name          (requires SuiNS config)
16. Get Name Task               (poll until confirmed)
17. Revoke Session              (sign out)
```
